Profiler yields per-function coverage, despite per-block level being set

Issue is reproducible with both Chrome DevTools UI and Chrome DevTools Protocol (called via Chrome extension)
DevTools API Reference

Important notes:

  1. Issue is not reproducible on _short_ (see short.html) scripts
    This page loads long.js script, which is >10k LOC

  2. Issue is not reproducible when page is served from filesystem
    Page has to be served from webserver (anything would fit, e.g. use SimpleHTTPServer if you have Python, or `npx serve` if you have Node.js installed )

  3. Issue is exactly reproducible with DevTools API (while Profiler.startPreciseCoverage({ detailed: true }) sets per-block level granularity, Profiler.takePreciseCoverage() yeilds per-function results, if conditions described in Reproduction steps are met

The script:

The script is simple. Only "foo()" function matters, the rest is a bloat. Second branch in if statement should never be executed, and thus - covered, when coverage type is set to per-block

function foo(x) {
if (x) {
console.log('branch 1')
} else {
console.log('branch 2') // THIS LINE _CANNOT_ BE COVERED
}
}

foo(true);
// ...bloat

Reproduction steps:

  1. Serve _this_ page (index.html) with webserver of your choice (ignore, if you opened this from a website)
  2. Open it
  3. Open DevTools
  4. Open "Sources" tab on top, and select "long.js"
  5. Open "Coverage" tab on the bottom
  6. Set granularity level to "Per-block" and click "record"
  7. Refresh the page once
  8. Note the correct results: 1. in capture results coverage type has "per-block" value 2. Line marked with comment is not covered
  9. Refresh the page one more time
  10. Note that results are unchanged
  11. Refresh the page last time
  12. THE ISSUE:
    1. coverage type field has "per-function" value (despite "per-block" is set in settings)
    2. Line marked with comment is covered now (this is impossible when per-block coverage is set)
  13. Interesting follow-up: if you do hard-refresh (CTRL+F5) the page, then coverage has "per-block" granularity again

What is happening/my guess

This issue completely vanishes, if you do hard-refresh each time you reload the page
It might be some issue with either how Chrome passes cached scripts to V8, or V8 itself handles cached/compiled scripts (if such cache is a thing)
